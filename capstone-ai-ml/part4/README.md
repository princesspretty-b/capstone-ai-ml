**LLM-Powered Feature: Structured Extraction, Tabular Batch Scoring, or Model Prediction Explanation**

**Choice of LLM Feature Track**

**Selected Track: C - Model Prediction Explanation Pipeline**

This feature adds an LLM-powered explanation layer on top of the trained diabetes classification model. The pipeline loads the best Random Forest model saved from Part 3, generates predictions and prediction probabilities for three manually created patient feature vectors, and sends the model output to an LLM API. The LLM returns a structured JSON explanation describing the prediction, confidence, main contributing factors, and recommended next steps.

**Task 1: Set up the LLM API connection**

**API Key Management**

The LLM API key is stored securely using an environment variable OPENROUTER_API_KEY. The API key is not stored in the Python files, or GitHub repository. It is stored in .env file in VS code. The Python code accesses the key using

os.getenv("OPENROUTER_API_KEY"). This prevents accidental exposure of credentials.

**LLM API Function Design**

A reusable function named call_llm was created. The function performs the following steps:

- Creates the API request URL.
- Builds the HTTP request headers
- Creates the JSON request body containing:

- model name
- system message
- user message
- temperature
- maximum output tokens

- Sends the request using requests.post
- Checks whether the API returned HTTP status code 200. If the request fails, the function prints the error status code and returns None. If successful, the function extracts the response using response.json()\['choices'\]\[0\]\['message'\]\['content'\] and returns the generated text.

**Test Prompt Demonstration**

A simple test request was used to send an API request and get response as hello which confirms that the LLM API connection is working correctly.

**Temperature Selection**

The initial API test used temperature=0. A temperature of zero was selected because structured machine learning tasks require consistent and reproducible outputs. With temperature=0:

- The model selects the highest probability token.
- Responses are more deterministic.
- JSON formatting errors are reduced.
- Repeated requests produce similar outputs.

Higher temperatures introduce randomness, which can be useful for creative tasks but may reduce reliability for structured prediction explanations.

**Why Temperature = 0 is chosen:**

A temperature value of **0** makes the model produce highly deterministic and consistent responses by selecting the highest-probability next token at each step. This is appropriate for structured JSON generation because the output format must remain stable and conform to the required schema for automated parsing and validation.

Using a higher temperature (for example, **0.7**) would increase randomness in the generated text. While this can produce more varied or creative explanations, it also increases the likelihood of inconsistent wording, additional text outside the JSON object, or deviations from the required schema. Since this project relies on machine-readable JSON responses that are validated programmatically, deterministic output is preferred.

Therefore, **temperature = 0** was chosen to maximize consistency, reproducibility, and reliable schema validation across all prediction explanations.

Temperature controls how the language model selects the next token during text generation. At **temperature = 0**, the model behaves deterministically by always selecting the highest-probability next token. As a result, repeated runs with the same input produce identical or nearly identical outputs, making the responses consistent and predictable. At **temperature = 0.7**, the model samples from a broader probability distribution instead of always choosing the single most likely token. This introduces controlled randomness, allowing different word choices, phrasing, or levels of detail while preserving the overall meaning of the response. In the examples above, the temperature 0 outputs were more consistent and standardized, whereas the temperature 0.7 outputs showed variations in labels (e.g., "Diabetic" versus "Positive for diabetes risk"), wording, and recommendation style without changing the underlying prediction.

**Task 4: Guardrails**

To improve privacy and prevent sensitive information from being sent to the LLM, a regex-based PII guardrail was implemented before every API call. The guardrail checks the user input for common email address and phone number patterns. If any PII is detected, the request is blocked and the LLM is not called.

The following function was used:

**def has_pii(text):**

**email*pattern = r'\[a-zA-Z0-9*.+-\]+@\[a-zA-Z0-9-\]+\\.\[a-zA-Z0-9-.\]+'**

**phone_pattern = r'\\b\\d{10}\\b|\\b\\d{3}\[-.\\s\]\\d{3}\[-.\\s\]\\d{4}\\b'**

**return bool(re.search(email_pattern, text) or**

**re.search(phone_pattern, text))**

If the function returns True, the program prints "Input blocked: PII detected." and skips the LLM request.
